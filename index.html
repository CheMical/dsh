Cross platform orchestration

Docker Shell lets you set up a development or staging environment inside a Docker container, which makes it relatively painless to use identical environments on many different platforms.

Features

Persistent workspace
Docker Shell sets up a persistent workspace stored inside a data container.
Experiment with new tools
Setting up extra tools inside Docker Shell is easy - only commit the changes that you're satisfied with.
Built in SMB support
Access to the workspace using SMB is supported.

Getting started

      git clone git@github.com:Solfisk/dsh
      dsh/dsh

You need to have permission to execute docker run.

Parameters

The synopsis for dsh is:

dsh [-b|--build] [-s|--samba] [--volumes-from <container>] [-v <hostpath>:<containerpath>] [-- <command> [ <parameters...> ] ]
-b | --build
(Re)build the docker shell image - rarely needed since this is done automatically on first run.
-s | --samba
Export workspace using SMB.
--volumes-from <container>
Will start Docker Shell with the volumes exported by <container>.
-v <hostpath>:<containerpath>
Will mount the <hostpath> inside Docker Shell at <containerpath>

Anything after two trailing dashes will be run as a command (and its parameters inside Docker Shell upon startup.

Examples

dsh -v ~/Downloads/assets:/tmp/assets -- rsync /tmp/downloads /workspace/my-project/assets

Synchronize ~/Downloads/assets to my-project inside the workspace.

dsh --volumes-from my-data-container -- ls /my-data-container

Look in the catalog /my-data-container provided by my-data-container (assuming this is exported by my-data-container).
